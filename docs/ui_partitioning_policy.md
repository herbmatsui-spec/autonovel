# UIフラグメント化設計方針 (UX/パフォーマンス最適化)

## 目的
Streamlit の再実行（Full Rerun）によるUIのラグを抑制し、ユーザーの操作性を劇的に改善する。

## 分割方針
アプリケーションを以下の4つの独立したフラグメントに分割する。

### 1. 進捗監視フラグメント (`@st.fragment`)
- **役割**: 生成プロセスの進捗状況（進捗バー、テキストストリーミング）の表示。
- **更新**: `BackgroundReporter` と同期し、ポーリングまたは SSE で更新。
- **親UI**: メインエリア上部に固定。

### 2. 入力フォームフラグメント (`@st.fragment`)
- **役割**: パラメータ入力（執筆スタイル、文字数等）のフォーム提供。
- **更新**: フォーム内で完結し、API呼び出しトリガー時にメインUIを再実行させない（`on_change` での状態更新に限定）。

### 3. 作品表示フラグメント (`@st.fragment`)
- **役割**: 生成された小説本文、メタデータの表示。
- **更新**: 生成完了時に外部からの通知を受けて更新。

### 4. サイドバーコントロールフラグメント (`@st.fragment`)
- **役割**: ナビゲーション、グローバル設定。
- **更新**: ほぼ更新されないため、静的なサイドバーとして固定。

## 通信設計
- 各Fragmentは `st.session_state` を介して状態を読み込む。
- Fragment間の直接的な関数呼び出しは禁止し、`UIStateStore` ( `streamlit_app/state.py` ) を介した Pub/Sub パターンを採用する。
